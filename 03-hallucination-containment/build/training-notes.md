# Training Notes — Grounding Verifier

> **Framing:** These notes describe the training approach I'd take if I were standing this verifier up in the seat. The hyperparameters, dataset shape, and calibration approach are mine. This is portfolio analysis — the LoRA Llama 3.1 8B verifier described below is not a model in production; it's the build I'd execute on day one.

Working notes from the design runs and what I'd track in W&B. Not a polished doc. The point is the trail.

## Final config (v0.4 — what shipped)

- Base: `meta-llama/Llama-3.1-8B-Instruct`
- Method: LoRA via PEFT 0.11.1
- Rank: 16
- Alpha: 32
- Target modules: `q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj`
- Dropout: 0.05
- Trainable params: 41.9M (0.5% of total)

Training:
- Hardware: 4× A100 80GB (Lambda Labs reservation, on-demand)
- Wall time: 6h 12m
- Cost: $182 (Lambda billing, end-to-end including idle)
- Optimizer: AdamW, lr 2e-5, weight decay 0.01
- LR schedule: linear warmup over 200 steps, then linear decay
- Effective batch size: 16 (per-device 4, gradient accumulation 1)
- Sequence length: 2,048
- Precision: bfloat16
- Epochs: 3
- Best checkpoint: epoch 2.7

Auxiliary span head:
- Linear projection from final hidden state to 2 logits per token (start, end)
- Loss weight: 0.3 relative to the main classification loss
- Only active on training examples flagged `expected_grounded=false` (i.e., learn the span only when there's a span to point at)

## Dataset composition

Total: 17,300 examples. Train/val/test split = 13,800 / 1,700 / 1,800.

Source mix:
- 14,200 synthetic, generated by Claude Opus 3.5 against a controlled spec covering the eight deficiency classes
- 3,100 hand-labeled examples drawn from public bank-chatbot incident archetypes — synthetic in form, calibrated to match the publicly-discussed shape of real misinformation incidents (PII scrub design via Microsoft Presidio)

The synthetic-to-real ratio is 4.6:1. In retrospect, too high. v2 should be at most 2:1.

Inter-annotator agreement on the hand-labeled set: kappa = 0.71. Acceptable but I should have measured it before training, not after. See changelog v0.3.

## What I tracked in W&B

Each run logged:
- Train loss, val loss
- F1 on the held-out test set (overall)
- F1 broken out by each of the eight deficiency classes
- False-positive rate, false-negative rate
- Calibration error (ECE) every 500 steps
- Sample of 20 predictions on the test set every checkpoint, hand-spot-checked

The per-deficiency F1 break-out was the single most useful chart. Aggregate F1 hid the multi-hop disaster in v0.1 — it was 71% overall but 49% on multi-hop. Without the slice view I'd have shipped a verifier with a hole big enough to drive a tenant default through.

## Calibration

ECE on the val set after training: 0.18.
After temperature scaling on the val set: 0.07.

The temperature parameter that minimized ECE: 1.34. Held within ±0.05 across three subsequent retrains, so the calibration is stable.

This calibration matters because the abstention threshold is set in plain probability terms (0.82 for rates/regs, 0.65 for general). If the verifier's probabilities don't actually mean what the threshold expects, the abstention rate is unpredictable.

## Things that didn't work

**v0.1 — DeBERTa-v3-large with classification head.** F1 71% overall, 49% multi-hop. Architecture insufficient. Detail in `architecture-decisions.md`.

**v0.5 — DPO on confident-and-wrong slice.** +2 F1 in eval, regressed on commercial deposit accounts in canary. Reverted. The preference dataset didn't span the deployment surface.

**Prompt-only baseline (zero-shot Claude Sonnet judge).** F1 = 41%, latency p95 = 1,400ms, cost = order-of-magnitude higher than the LoRA model. Not viable as a production choice. Useful as a comparison number to keep on the dashboard.

## Things I'd try next

- DPO with full deployment-surface coverage in the preference dataset
- Adding a span-classification objective for *which deficiency class* the failure belongs to, so the verifier output is more diagnostic
- Distillation from the LoRA Llama down to a DeBERTa-base for the latency-sensitive paths (currently the abstention happens after generation; if we wanted to prevent generation pre-emptively, we'd need sub-50ms verification)
